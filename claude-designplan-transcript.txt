⏺ 64/64 tests pass. No bugs found. Two potential issues were identified during
  code review and verified as non-issues:

  1. useEffect dependency on dueSoonTasks.length instead of the full array
  This is intentional and correct. Using the full dueSoonTasks array as a
  dependency would cause the effect to re-run every render (new array reference
  each time). Using .length means notifications only re-trigger if the count of
  due-soon tasks actually changes — which is the desired "once per session"
  behavior, guarded further by notifiedRef.

  2. isDueSoon called with an empty/missing dueDate
  If a task has dueDate: '', new Date('' + 'T00:00:00') produces Invalid Date.
  Comparisons against Invalid Date always return false, so the task is safely
  excluded from the banner. No crash, no unintended inclusion — confirmed by the

⏺ 75/75 pass. Here's what was changed and why:

  Root cause: <input type="date"> on mobile browsers (especially WebKit/Safari
  on iOS) injects its own internal date-picker chrome — spinner segments, a
  calendar icon — that ignores width: 100% and overflows its container.

  Fixes applied in App.css:

  1. Added box-sizing: border-box to all form inputs (ensures padding is
  included in the declared width, not added on top)
  2. Added a dedicated input[type="date"] rule:
    - display: block + width: 100% + max-width: 100% + min-width: 0 — forces the
   element to respect its flex/grid container bounds
    - appearance: none / -webkit-appearance: none — strips Safari's native date
  styling so the browser respects our sizing rules
    - padding-right: 10px — gives the browser's internal calendar icon room
  without squeezing the text
  3. Inside the ≤640px mobile breakpoint, reduced padding-right further to 6px
  and set font-size: 0.95rem so the date text segments fit cleanly on narrow
  screens
