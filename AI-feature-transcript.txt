  ---
  How it works

  Session persistence — AuthContext calls supabase.auth.getSession() on mount
  and subscribes to onAuthStateChange. Supabase stores the JWT in localStorage
  automatically, so sessions survive page refreshes.

  Protected routes — ProtectedRoute checks user from context. If null, it
  redirects to /login. If still loading, it renders nothing (avoids a flash).

  Guest routes — GuestRoute does the inverse: if the user is already logged in
  and visits /login or /signup, they're redirected to /.

  Sign-up flow — Supabase can require email confirmation (default) or not. The
  SignUp component checks data.session — if present, it navigates directly to
  the app; if not, it shows a "check your email" message. You can toggle this in
   your Supabase dashboard under Authentication → Email.

  ---
  One thing to do in your Supabase dashboard

  Go to Authentication → Providers → Email and decide whether to enable "Confirm
   email". If you want users to log in immediately after signing up (easier for
  testing), turn it off.

  ---
  Part 1 — SQL to run in your Supabase SQL Editor

  Go to Supabase dashboard → SQL Editor and run this:

  -- Step 1: Set user_id to auto-fill from the authenticated JWT
  --         This means you never have to send user_id from the frontend
  ALTER TABLE tasks
    ALTER COLUMN user_id SET DEFAULT auth.uid();

  -- Step 2: Enable Row Level Security on the table
  ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

  -- Step 3: SELECT — users can only read their own rows
  CREATE POLICY "select_own_tasks"
    ON tasks FOR SELECT
    USING (auth.uid() = user_id);

  -- Step 4: INSERT — users can only insert rows where user_id matches them
  --         USING is the read filter, WITH CHECK is the write filter
  CREATE POLICY "insert_own_tasks"
    ON tasks FOR INSERT
    WITH CHECK (auth.uid() = user_id);

  -- Step 5: UPDATE — both clauses needed:
  --         USING prevents reading rows they don't own to begin with
  --         WITH CHECK prevents changing user_id to someone else's
  CREATE POLICY "update_own_tasks"
    ON tasks FOR UPDATE
    USING (auth.uid() = user_id)
    WITH CHECK (auth.uid() = user_id);

  -- Step 6: DELETE — users can only delete their own rows
  CREATE POLICY "delete_own_tasks"
    ON tasks FOR DELETE
    USING (auth.uid() = user_id);

  If your table also needs category, description, and created_at — run this too:
  ALTER TABLE tasks
    ADD COLUMN IF NOT EXISTS category     text,
    ADD COLUMN IF NOT EXISTS description  text,
    ADD COLUMN IF NOT EXISTS created_at   timestamptz NOT NULL DEFAULT now();

  ---
  How auth.uid() works

  auth.uid() is a PostgreSQL function built into Supabase. When the frontend
  makes a request, the Supabase JS client automatically attaches the user's JWT 
  as a Bearer token in the Authorization header. Postgres reads that token,
  extracts the sub claim (which is the user's UUID), and that's what auth.uid()
  returns inside any policy or default expression.

  Frontend request
    └── Authorization: Bearer <JWT>
          └── Postgres reads JWT → auth.uid() returns user's UUID
                └── USING (auth.uid() = user_id) → only their rows pass

  Because user_id now defaults to auth.uid(), you never send user_id from the 
  frontend — the database fills it in automatically and the INSERT policy
  enforces it matches.

⏺ Clean. Here's a complete summary of how everything fits together:

  ---
  How RLS + the frontend work together

  The request flow for every operation

  useTasks calls supabase.from('tasks')...
    │
    └── Supabase JS attaches the user's JWT automatically
          │
          └── Postgres evaluates auth.uid() from that JWT
                │
                ├── SELECT  → USING (auth.uid() = user_id)   ← filters rows at
  DB level
                ├── INSERT  → WITH CHECK (auth.uid() = user_id) + DEFAULT
  auth.uid()
                ├── UPDATE  → USING + WITH CHECK              ← can't touch
  other users' rows
                └── DELETE  → USING (auth.uid() = user_id)   ← same guard

  Why you never send user_id from the frontend

  With DEFAULT auth.uid() on the column, Postgres fills it in from the JWT. The
  INSERT policy then double-checks it matches — so even if someone tried to
  forge a different user_id in the request body, Postgres would reject it. You
  get security at the database level, not just the API level.

  The toClient mapping

  Your DB uses snake_case (due_date, created_at); your UI uses camelCase
  (dueDate, createdAt). The toClient function in useTasks.js translates once at
  the boundary so nothing else in the app has to think about it.

  Optimistic vs. confirmed updates

  The hook uses confirmed updates — it only updates local state after Supabase
  responds without an error. This keeps the UI truthful about what's actually
  persisted.

⏺ Clean. Here's a full explanation of every decision made:

  ---
  What changed and why

  1. AuthContext.js — removed getSession()

  onAuthStateChange fires a synthetic INITIAL_SESSION event on mount that
  resolves the persisted session from localStorage. Calling getSession()
  separately before it was redundant — it triggered two separate state updates
  on startup with no benefit.

  Before:  getSession()  →  setUser + setLoading(false)
           onAuthStateChange fires → setUser again (double update)

  After:   onAuthStateChange INITIAL_SESSION fires once → setUser +
  setLoading(false)

  ---
  2. ProtectedRoute.js — spinner + location state

  loading === true   →  <AuthSpinner />     (was: null → blank screen flash)
  user === null      →  <Navigate to="/login" state={{ from: location }} replace
   />
  user exists        →  render children

  Passing state={{ from: location }} saves the full location object (pathname +
  search + hash) into navigation state. This is the standard React Router
  pattern for "redirect after login". The replace flag prevents the login
  redirect from polluting browser history.

  ---
  3. GuestRoute.js — new dedicated file

  Same mirror logic to ProtectedRoute:

  loading === true   →  <AuthSpinner />     (prevents flash of login page for
  logged-in users)
  user exists        →  <Navigate to={from} replace />   where from =
  state?.from?.pathname ?? '/'
  user === null      →  render children (show the auth page)

  The from fallback chain: if the user navigated directly to /login (no from
  state), they go to / after logging in. If they were bounced from /calendar,
  they go back to /calendar.

  ---
  4. Login.js — reads from and redirects there

  const from = location.state?.from?.pathname ?? '/';
  // ...
  navigate(from, { replace: true });

  replace: true replaces the login entry in history so the user can't press Back
   and return to the login page after signing in.

  ---
  The full redirect flow

  User visits /calendar (unauthenticated)
    └── ProtectedRoute: no user → Navigate to="/login" state={{ from: /calendar
  }}
          └── GuestRoute: loading=true → spinner, then renders Login
                └── Login: user submits form → navigate('/calendar', { replace:
  true })
                      └── ProtectedRoute: user exists → renders AppShell ✓

  User visits /login (already authenticated)
    └── GuestRoute: user exists → Navigate to="/" replace
          └── AppShell renders immediately ✓

  Page hard-refresh (session in localStorage)

          └── INITIAL_SESSION fires → user set, loading=false

⏺ Clean. Here's a full walkthrough of every decision:

  ---
  How the Supabase ↔ auth ↔ tasks connection works end-to-end

  Why user_id is never sent from the frontend

  INSERT payload from useTasks:
    { title, category, priority, due_date, description, completed }
    ← no user_id

  Postgres receives it:
    user_id column DEFAULT auth.uid()  → fills from the JWT automatically
    RLS INSERT policy: WITH CHECK (auth.uid() = user_id)  → verifies it

  If someone crafted a raw request with a forged user_id, the RLS policy would
  reject it at the database. The protection is server-side, not just
  client-side.

  ---
  Why the fetch has no .eq('user_id', user.id) filter

  supabase.from('tasks').select('*').order('created_at', { ascending: false })

  There's no explicit WHERE user_id = ? — RLS handles it. The Supabase client
  attaches the user's JWT as a Bearer token on every request, Postgres calls
  auth.uid() from that token, and the SELECT policy silently filters the result
  set. You cannot accidentally fetch another user's rows.

  ---
  The three state lifecycle problems that were fixed

  1. Silent failures → surfaced errors

  Before:
  if (!error) setTasks(data.map(toClient));
  // error is silently discarded

  After: all three operations (addTask, deleteTask, toggleComplete) set
  taskError and return { error } so callers can react if needed.

  2. Stale tasks on logout

  Before: when user became null, the effect returned early — tasks stayed in
  memory.

  After:
  if (!user) {
    setTasks([]);        // wipe immediately
    setTasksLoading(false);
    return;
  }

  If two users share a browser session (sign out → sign in as someone else),
  User B never sees User A's tasks.

  3. tasksLoading never reset between sessions

  Before: tasksLoading started true, was set false once, and never reset — so if
   a user logged out and back in within the same component lifetime, the second
  fetch had no loading state.

  After:
  setTasksLoading(true);   // reset at the top of every fetch
  setTaskError(null);

  ---
  State model at each point in time

  ┌─────────────────────────────┬──────────────┬────────────────┬───────────┐
  │          Scenario           │ tasksLoading │     tasks      │ taskError │
  ├─────────────────────────────┼──────────────┼────────────────┼───────────┤
  │ App cold start, session     │ true         │ []             │ null      │
  │ loading                     │              │                │           │
  ├─────────────────────────────┼──────────────┼────────────────┼───────────┤
  │ Logged in, fetch in flight  │ true         │ []             │ null      │
  ├─────────────────────────────┼──────────────┼────────────────┼───────────┤
  │ Fetch succeeded             │ false        │ [...rows]      │ null      │
  ├─────────────────────────────┼──────────────┼────────────────┼───────────┤
  │ Fetch failed                │ false        │ []             │ "message" │
  ├─────────────────────────────┼──────────────┼────────────────┼───────────┤
  │ User logs out               │ false        │ []             │ null      │
  ├─────────────────────────────┼──────────────┼────────────────┼───────────┤
  │ Second user logs in         │ true → false │ [...their      │ null      │
  │                             │              │ rows]          │           │
  └─────────────────────────────┴──────────────┴────────────────┴───────────┘

  ---
  Confirmed vs optimistic updates

  All three write operations use confirmed updates — local state only changes
  after Supabase responds without an error:

  addTask("Buy milk")
    → supabase.insert(...)        wait for DB...
    → { data: { id: 'abc', ... }, error: null }
    → setTasks(prev => [toClient(data), ...prev])   ✓ DB id used, not a temp id

  The trade-off: there's a brief delay between clicking and seeing the new task.
   The benefit: the task in state has the real DB-generated id and created_at,
  so deletes and toggles immediately work correctly with .eq('id', id).
  Optimistic updates would require rollback logic on failure — not worth the
  complexity here.
